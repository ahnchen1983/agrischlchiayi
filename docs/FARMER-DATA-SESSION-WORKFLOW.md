## 📌 核心原則

### 1. Session 獨立
- 每篇文章的框架已在 FARMER-DATA-EXECUTION-SCHEDULE.md 中準備
- Session 中斷後，無縫繼續（只需查看計劃文檔）
- 不依賴 Session 記憶，所有信息都在文檔中

### 2. 去識別化第一
- 在發布前必須通過「去識別化驗證清單」
- 每篇文章都需要人工審查
- 疑問時寧可刪除也不保留

### 3. 進度透明
- 更新執行計劃的進度追蹤
- 記錄完成日期、驗證日期、備註
- 每個 commit 包含驗證狀態

### 階段 1：內容撰寫（1-3 小時）

#### 步驟 1.1：提取農友數據
```
對於每個文章指定的農友（如 3-5 位）：

1. 從 source/ 目錄找到農友 MD 檔案
2. 提取以下信息：
   ✓ 作物類型、規模（公頃）
   ✓ 年產量、年收入
   ✓ 銷售通路和比例
   ✓ 主要技術和設備
   ✓ 常見問題和解決方案
   ✓ 與其他農友的關聯（Wikilinks）

3. 記錄農友代號（如 F1、F2、F3）便於追蹤

提取格式範例：
  農友 F1（Farmer-C）:
    - 作物：小黃瓜 0.3 ha
    - 年產：6 公噸
    - 通路：行口 55%、批發 17%、直銷 28%
    - 特色：三作物輪作、家庭農業
    - 關鍵技術：基礎設施農業、人工管理
```

#### 步驟 1.2：撰寫文章框架
```
按照計劃文檔中的「文章框架」撰寫：

1. Frontmatter（確保完整）
   - title, description
   - category, crop, tech, level
   - target, season, keywords

2. 按章節順序撰寫內容
   - 每章有明確的「農友案例」段落
   - 每個案例使用去識別化代號（F1、F2、F3）

3. 添加實務表格和清單
   - 月份計劃表
   - 成本分析
   - 常見問題 Q&A
   - 檢查清單

4. 包含 Wikilinks
   - 相關文章使用 [[標題]] 格式
```

#### 步驟 1.3：第一次自檢
```
撰寫完成後，自己檢查：

[ ] 是否有農友的真實姓名出現？
[ ] 是否有詳細地址（郵編、村里號）？
[ ] 是否有聯絡方式（電話、郵箱、社交媒體）？
[ ] 是否有家庭成員名字？
[ ] 表格和數據是否準確無誤？
[ ] Frontmatter 是否完整？
[ ] Wikilinks 是否都存在？
```

### 階段 2：去識別化驗證（30-60 分鐘）

#### 核心驗證流程

這是**最關鍵的階段**。執行以下檢查清單：

#### ✅ 檢查清單 A：直接標識符移除

```
【農友姓名】
檢查方法：搜尋 "name", "姓名", "農友"
- [ ] 文中無出現任何農友真實姓名
- [ ] 農友通用代稱為「某農友」或「某位農民」
- [ ] 如有多位農友，用「農友 A」「農友 B」或「設施農民」「蔬菜農友」等通用語

案例對比：
❌ 錯誤：「Farmer-C在東石鄉種小黃瓜...」
✅ 正確：「某位東石鄉設施小黃瓜農友...」或「某設施農民（0.3 ha）...」

【詳細地址】
檢查方法：搜尋 "里", "村", "號", "路", "鎮"
- [ ] 無出現完整郵編（如 613-26）
- [ ] 無出現具體村里號（如「東石鎮三家村 123 號」）
- [ ] 可保留「東石地區」「嘉義縣」這樣的廣域級別
- [ ] 如提及鄉鎮，最多到「鄉/鎮」級別，不含村里

案例對比：
❌ 錯誤：「朴子市朴子村 5 鄰 88 號」
✅ 正確：「朴子地區」或「朴子一帶」

【聯絡方式】
檢查方法：搜尋 "@", "電話", "手機", "Line", "Facebook"
- [ ] 無出現電話號碼（完整或部分）
- [ ] 無出現 Email 地址
- [ ] 無出現社交媒體帳號（Facebook、Line、Instagram）
- [ ] 無出現 Line ID 或微信號
```

#### ✅ 檢查清單 B：間接標識符移除

```
【家庭成員信息】
檢查方法：搜尋 "太太", "老婆", "先生", "媽媽", "父", "子"
- [ ] 無出現家庭成員的名字
- [ ] 無出現具體家庭結構（「太太務農，兒子在台北」）
- [ ] 可保留「家庭農業」「多代經營」等通用描述

案例對比：
❌ 錯誤：「Farmer-F的太太李雨晴和他一起務農」
✅ 正確：「某農友與家人共同經營」或「家庭農場」

【獨特的工作經歷】
檢查方法：搜尋 "原本", "曾經", "之前", "背景"
- [ ] 如提及「原為台北藥廠業務」這類獨特背景，要模糊化
- [ ] 改為「原從事其他產業」或「有非農業工作背景」
- [ ] 如果農友背景獨特到可被識別，考慮刪除或高度模糊化

案例對比：
❌ 錯誤：「某農友原為台北大型製藥廠的業務經理...」（可能被農友本人識別）
✅ 正確：「某位有業務背景的返鄉農民...」或完全刪除此背景

【特殊設備或技術】
檢查方法：搜尋 "Autopot", "某品牌", "獨家技術"
- [ ] 農友採用的特殊設備或專利技術，如果太獨特，考慮模糊化
- [ ] 如「Autopot 水耕系統」是該農友的獨特特徵，可用「某類進口水耕系統」代替
- [ ] 保留通用技術描述（「滴灌」「環控」）

案例對比：
❌ 錯誤：「採用張[Farm-B]老師研發的智能養液設備...」（指向特定人物）
✅ 正確：「採用先進的智能養液控制系統...」

【特殊認證或獎項】
檢查方法：搜尋 "獲獎", "認證", "卓越"
- [ ] 如農友有特殊認證（有機、產銷履歷 PLUS 等），可保留
- [ ] 但不要提「第 X 屆卓越班」這類具體課程信息
- [ ] 改為「經過農業培訓」或「持有農業認證」

案例對比：
❌ 錯誤：「第四屆國本學堂卓越班學員...」（指向特定班級）
✅ 正確：「經過農業進階培訓的農民...」或「有農業認證的農友...」
```

#### ✅ 檢查清單 C：「反向工程測試」

這是**最重要的測試**。想像您是網路上的陌生人，讀了這篇文章：

```
問題 1：我能從這篇文章推測出「具體農民」嗎？
  ✓ 如果答案是「不能」→ 去識別化成功
  ✗ 如果答案是「能」→ 需要進一步模糊化

問題 2：農民親戚或鄰居能從這篇文章認出他嗎？
  ✓ 如果答案是「不能」→ 去識別化成功
  ✗ 如果答案是「很可能」→ 刪除具體細節

問題 3：這篇文章中是否有「2-3 個農友」的集合特徵？
  ✓ 如果是 → 實現了去識別化的核心目標
  ✗ 如果不是，只是 1 個農友的故事 → 需要更多農友案例混合
```

#### 具體驗證範例

假設您寫了這段：

```
「某設施農民（0.3 公頃，年產 6 公噸小黃瓜）在設施農業中
通過精準肥水管理，達成年收 180 萬元。通路以行口（55%）
和批發（17%）為主，直銷占 28%。採用三作物輪作策略
（番茄 + 美濃瓜 + 小黃瓜），以降低單一作物風險。」
```

**驗證過程**：
```
1. 有沒有姓名？ → 沒有 ✓
2. 有沒有地址？ → 只有「設施農業」，沒有具體位置 ✓
3. 有沒有家庭信息？ → 沒有 ✓
4. 能推測出誰嗎？
   - 「0.3 ha 小黃瓜」— 多位農友符合，不唯一
   - 「年收 180 萬」— 範圍廣，不唯一
   - 「行口 55%、批發 17%、直銷 28%」— 特定比例，有識別風險
   - 「三作物輪作」— 組合很獨特，可能被識別
   
   → 建議進一步模糊化：改為「多通路銷售，其中批發與中間商為主」
   → 改為「通過作物輪作」而非具體列舉三種作物
```

## 🔄 Session 工作流程

### 開始一個新 Session：

```
1. 打開此文檔 (FARMER-DATA-SESSION-WORKFLOW.md)
2. 打開執行計劃 (FARMER-DATA-EXECUTION-SCHEDULE.md)
3. 找到「進度追蹤」，查看最後完成的文章
4. 找到下一篇待開始的文章
5. 按此文檔「階段 1-3」流程執行
```

### Session 中斷時：

```
1. 記下當前進度（%）
2. 在執行計劃中標記
3. 下次 Session：
   - 打開執行計劃
   - 找到上次的文章
   - 繼續執行（從哪裡斷掉就從哪裡接上）
```

### 完成一篇文章：

```
1. 執行階段 2 去識別化驗證
2. 填寫「去識別化驗證報告」
3. git add knowledge/.../文章.md
4. git commit -m "content: add [文章名] with [X] farmer case studies

   去識別化驗證：✅ 通過
   農友數據源：[列出農友]
   風險等級：低"
5. 在執行計劃中更新進度
6. git add docs/FARMER-DATA-EXECUTION-SCHEDULE.md
7. git push
```

## ⚠️ 常見去識別化錯誤與修正

| 錯誤例子 | 為什麼有問題 | 正確做法 |
|---------|-----------|---------|
| 「Farmer-C用三作物輪作...」| 直接出現姓名 | 「某農友採用多作物輪作...」|
| 「朴子市朴子村 123 號的農民...」| 詳細地址可定位 | 「朴子地區的農民...」|
| 「這位農友原為台北大型製藥廠業務...」| 背景太獨特，可被認出 | 「某位有業務背景的農民...」或刪除 |
| 「採用 Autopot 系統...」| 品牌太獨特 | 「採用進口自動潮汐灌溉系統...」|
| 「第四屆卓越班第 X 號...」| 課程編號可被追蹤 | 「經過農業進階培訓...」|
| 「和太太李XX一起管理農場」| 配偶名字洩露身份 | 「與家人共同經營...」|
| 表格顯示「年收 210 萬、銷售比例 49%-36%-14%」| 組合太獨特，易被識別 | 模糊化為「年收 200 萬上下」「以批發和中間商為主」|

## 🎯 去識別化的終極原則

```
「讀者能從文章中學到技術，但無法推測出具體農民。」

這就是成功的去識別化。

如果你對某句話有疑問：
  「這會不會讓農友被認出？」

答案是「可能」→ 刪除或模糊化
答案是「不太可能」→ 可以保留

寧可謹慎，也不要冒風險。
```

---

## 📞 驗證疑問時的決策樹

```
遇到一句話，不確定是否符合去識別化？

問 1：這句話是否包含「農友可被直接識別的信息」？
  → 是：刪除或改為通用描述
  → 否：進行問 2

問 2：這句話結合其他信息，是否會「幫助認出農友」？
  → 是：刪除或進一步模糊化
  → 否：進行問 3

問 3：讀者是否必須知道這句話才能學到技術？
  → 是：保留，但嘗試改為通用描述
  → 否：刪除

最終原則：
  有疑問 → 刪除或模糊化
  沒疑問 → 保留
```

## 範例驗證過程

### 文章段落範例

```
「某位東石地區的設施蔬菜農友（小黃瓜 0.3 公頃，
年產 6 公噸），通過精準的肥水管理和環控系統，
實現年收 180 萬元。其銷售策略以行口（55%）和
批發市場（17%）為主，直銷占 28%。」
```

### 驗證過程
