## 場景 1：新農友資料進來

### 何時觸發？

- 🔔 收到新農友的經營計畫書
- 🔔 農民提供業務資料或案例故事
- 🔔 農會或政府機構提供農民名單

### 完整流程

```
新農友資料進來
    ↓
【命令】召喚我執行 FARMER-DATA-PROCESSING-CHECKLIST
    ↓
你說：「新農友進來了，我有一份農友資料要處理」
    ↓
我會：
  1. 打開 FARMER-DATA-PROCESSING-CHECKLIST.md
  2. 提取基礎信息（作物、規模、問題）
  3. 對齐分類（用 CATEGORY-MAPPING-GUIDE.md）
  4. 驗證去識別化（用 DEIDENTIFICATION-CHECKLIST.md）
  5. 決定行動（補充 vs 新建）
  6. 執行發布
    ↓
完成
```

### 你應該告訴我什麼？

**必填信息**：
```
新農友資料：[農友的經營計畫書或關鍵信息]

其中包括：
  - 作物：_________
  - 規模：_________
  - 銷售通路：_________
  - 核心問題：_________
  - 年收入：_________
  - 其他重要信息：_________
```

**或簡單說**：
```
「我有一份小番茄農友的資料要整理」
「農友規模 0.5 公頃，年收 150 萬，想做直銷」
```

### 例子

你說：
> 新農友資料進來了。他種小番茄 0.25 公頃，年收 80 萬，目前全部賣給行口，想轉直銷但不知道怎麼做。

我會：
1. 識別：小番茄 + 直銷困難 + Agri-Marketing
2. 檢查：是否有「直銷客戶開發實務」文章？
3. 決定：補充現有文章（而非新建）
4. 執行：在「直銷客戶開發實務」中添加去識別化案例

## 場景 2：整理現有內容

### 何時觸發？

- 📋 想重新整理全站文章
- 📋 發現某個分類混亂或不符合標準
- 📋 想優化知識庫結構

### 完整流程

```
整理內容需求
    ↓
【命令】召喚我執行 CONTENT-RESTRUCTURING-PLAN
    ↓
你說：「現有內容需要整理，符合新的架構」
    ↓
我會：
  1. 逐個檢視 36 篇文章
  2. 確認是否符合分類標準
  3. 識別需要修改的文章（標題、分類、去向）
  4. 生成整理清單
  5. 執行修改（逐步）
    ↓
完成
```

### 你應該告訴我什麼？

**簡單說**：
```
「把現有的 36 篇文章整理一下，符合新架構」
```

或指定範圍：
```
「Farm-Management 這個分類有點亂，幫我整理一下」
```

### 例子

你說：
> 我們現在有清晰的 10 大分類架構，需要把現有的 36 篇文章都重新整理。檢查是否符合標準，該調整的標題、該移出的也要處理。

我會：
1. 逐個檢視 36 篇文章
2. 生成「優先級清單」（需要修改什麼）
3. 建議執行步驟
4. 一步步執行（獲得你的確認）

## 場景 3：驗證去識別化

### 何時觸發？

- 🔒 新文章寫好了，要發布前檢查
- 🔒 現有文章發現有隱私風險
- 🔒 定期審核已發布文章

### 完整流程

```
文章寫好 / 發布前
    ↓
【命令】檢查去識別化
    ↓
你說：「幫我檢查這篇文章有沒有隱私問題」
    ↓
我會：
  1. 逐段檢查（開場、案例、數據）
  2. 用 DEIDENTIFICATION-CHECKLIST 驗證
  3. 標記有問題的地方
  4. 提出修改建議
    ↓
完成 / 通過發布
```

### 你應該告訴我什麼？

**貼上文章內容**或**給檔案路徑**：

```
「檢查這篇文章：knowledge/Agri-Marketing/[文章名].md」

或

「檢查去識別化：[貼上內容]」
```

### 例子

你說：
> 我寫好了一篇「新農友直銷案例」的文章，貼在下面，幫我檢查有沒有洩露個人隱私。

[貼文章內容]

我會：
1. 逐段掃描（找農民姓名、地址、電話）
2. 檢查數據是否太具體（精確年收 → 改為範圍）
3. 標記可改進之處
4. 給出修改建議

## 場景 4：發布新文章

### 何時觸發？

- 📤 新文章寫好，確認無誤
- 📤 要同步、構建、推送到遠端

### 完整流程

```
文章完成 + 通過檢查
    ↓
【命令】發布新文章
    ↓
你說：「新文章完成了，幫我同步和發布」
    ↓
我會：
  1. 檢查 Frontmatter 完整性
  2. 檢查 Wikilink 正確性
  3. 運行 npm run sync（同步到發布層）
  4. 運行 npm run build（構建網站）
  5. 提交 Git commit
  6. 推送遠端（git push）
    ↓
完成（文章上線）
```

### 你應該告訴我什麼？

**簡單說**：
```
「文章完成了，發布吧」
```

或指定文章：
```
「發布：knowledge/Agri-Marketing/新農友案例.md」
```

## 常見問題

### Q1：如果我說「新農友進來了」，你怎麼知道要做什麼？

**A**：我會自動執行 FARMER-DATA-PROCESSING-CHECKLIST 的 6 步流程：
1. 提取基礎信息
2. 分類對齐
3. 去識別化驗證
4. 內容品質檢查
5. 決定行動
6. 最終發布

你只需要提供農友的基本信息，剩下的交給我。

---

### Q2：如果文章太特殊，不符合現有分類怎麼辦？

**A**：根據 CATEGORY-MAPPING-GUIDE，有三種處理方式：

1. **情況 A**：補充現有文章（大多數）
2. **情況 B**：創建新文章（只在以下情況）
   - 新作物（嘉義沒有相關文章）
   - 新通路或新技術
   - 新產業方向
   - **且** 符合「3 農友規則」（3+ 農友有同樣問題）
3. **情況 C**：保存待決（目前不處理，等更多案例）

我會根據上述標準決定最佳行動。

---

### Q3：發布流程具體是什麼？

**A**：四個步驟：

```bash
1. npm run sync    # 同步 knowledge/ → src/content/
2. npm run build   # 構建網站和搜尋索引
3. git commit      # 提交變更
4. git push        # 推送到遠端
```

你說「發布」，我自動執行這四步。

---

### Q4：整個流程文檔在哪裡？

**A**：在 `/docs/` 目錄下，共 4 份核心文檔：

1. **CATEGORY-MAPPING-GUIDE.md**

4. **CONTENT-RESTRUCTURING-PLAN.md**
   - 現有 36 篇文章的整理計畫

## 使用建議

### ✅ 推薦做法

1. **明確告訴我觸發信號**
   ```
   ✅「新農友進來了」
   ✅「農友資料需要處理」
   ✅「整理現有內容」
   ```

2. **提供完整信息**
   ```
   ✅「新農友：小番茄農民，0.25 公頃，年收 80 萬，想做直銷」
   ✅「文章完成：knowledge/Agri-Marketing/新文章.md」
   ```

3. **分步確認**
   ```
   ✅ 第一步：我提出整理清單
   ✅ 第二步：你確認優先順序
   ✅ 第三步：我執行修改
   ```

### ❌ 避免做法

1. ❌ 只說「幫我寫文章」（太模糊，我不知道是新文章還是補充現有）
2. ❌ 不提供農友信息（我無法開始處理）
3. ❌ 期望我自動知道優先順序（請明確說明）
