# 農友資料處理檢查清單（Farmer Data Processing Checklist）

**版本**: 1.0  
**日期**: 2026-04-19  
**用途**: 當新農友資料進入時，用於確保品質、去識別化、分類對齊的標準化流程

---

## 📋 農友資料處理流程（6 步）

### 第 1 步：基礎信息提取（Data Extraction）

接收新農友資料後，立即提取關鍵信息：

**作物信息**
- [ ] 主要作物（可複選）：_________________
- [ ] 副業作物：_________________
- [ ] 新導入品種（若有）：_________________

**經營規模**
- [ ] 總面積：_____ 公頃
- [ ] 其中設施面積：_____ 公頃（若有）
- [ ] 年產量：_____ 噸/年

**銷售渠道**
- [ ] 主要通路：☐ 批發 ☐ 行口 ☐ 直銷 ☐ 電商 ☐ 超商 ☐ 出口
- [ ] 各通路比例：批發____% 行口____% 直銷____% 其他____% 
- [ ] 年收入估計：_____ 萬元

**核心問題**（農友主要想解決什麼）
- [ ] 生產問題：□ 品種 □ 栽培技術 □ 病蟲害 □ 高溫/乾旱
- [ ] 經營問題：□ 成本控制 □ 人力管理 □ 財務困難
- [ ] 銷售問題：□ 通路困難 □ 價格低 □ 客戶開發 □ 品牌建立
- [ ] 設施問題：□ 溫室老化 □ 環控系統 □ 灌溉不足
- [ ] 技術導入：□ 想用 IoT □ 無人機 □ 其他新技術

**其他重要信息**
- [ ] 經驗年限：_____ 年
- [ ] 是否有員工：□ 無 □ 1-2 人 □ 3-5 人 □ 5 人以上
- [ ] 是否已有品牌/商標：□ 無 □ 有（_________）
- [ ] 認證狀態：□ 無 □ 有機 □ 友善 □ 產銷履歷 □ 其他

---

### 第 2 步：分類對齊（Category Alignment）

根據基礎信息，將農友問題對應到 10 大分類：

**作物相關 → Crop-Production（農作物生產技術）**
- [ ] 農友有「栽培技術」問題 → 檢查是否有該作物的文章
  - 若有：補充農友的案例
  - 若無：列入「內容缺口」，規劃新文章
- [ ] 農友有「病蟲害」問題 → 補充或創建該作物的診斷指南

**設施相關 → Facility-Farming（設施農業）**
- [ ] 農友提及「溫室老化/需要重建」 → 補充設施規劃文章
- [ ] 農友提及「環控系統」 → 補充或創建環控指南

**銷售相關 → Agri-Marketing（農產品銷售與品牌）**
- [ ] 農友有「通路困難」 → 補充銷售通路策略
- [ ] 農友有「客戶開發」問題 → 補充直銷客戶開發指南
- [ ] 農友有「品牌建立」需求 → 補充品牌策略文章
- [ ] 農友有「出口意圖」 → 補充國際市場文章

**經營相關 → Farm-Management（農場經營與管理）**
- [ ] 農友有「財務困難」 → 補充財務規劃文章
- [ ] 農友有「人力管理」問題 → 補充人力管理文章

**技術相關 → Smart-Farming（智慧農業）**
- [ ] 農友想「導入 IoT/自動灌溉」 → 補充智慧農業文章
- [ ] 農友想「用無人機」 → 補充無人機應用文章

**其他問題**
- [ ] 補助/資金 → Grants-Planning
- [ ] 動物養殖 → Livestock-Health
- [ ] 基礎知識不足 → Agri-Basics 或 Agri-Advanced

**分類結果**
```
農友問題對應：
  1. _____________ → [分類]
  2. _____________ → [分類]
  3. _____________ → [分類]
```

### 第 3 步：去識別化驗證（De-identification Verification）

**必須移除的信息**：
- [ ] ❌ 農民姓名（全名或常用名）
- [ ] ❌ 農場名稱（若能識別到個人）
- [ ] ❌ 具體地址（鄉鎮以下）
- [ ] ❌ 電話號碼、Line ID、Email
- [ ] ❌ 家庭成員名字
- [ ] ❌ 獲獎記錄/公開活動名單

**可以保留的信息**：
- [ ] ✅ 作物類型
- [ ] ✅ 經營規模（「0.35 公頃」）
- [ ] ✅ 銷售通路比例
- [ ] ✅ 年收入估計數字
- [ ] ✅ 成本結構
- [ ] ✅ 技術方案
- [ ] ✅ 經營理念和故事

**去識別化範例**：

❌ 原文：「Farmer-D在六腳鄉種小黃瓜，年收約 200 萬，他最近買了智慧灌溉系統...」

✅ 修正：「某中型設施小黃瓜農場（0.35 公頃，年產 75 噸），最近導入智慧灌溉系統...」

---

### 第 4 步：內容品質檢查（Content Quality Check）

#### 文章標題和描述
- [ ] 標題清晰、無課程編號（如「第四屆...」）
- [ ] 描述簡潔（一句話說明核心內容）
- [ ] 標題避免超過 60 字

#### 結構和邏輯
- [ ] 有清晰的開場段（核心問題或洞見）
- [ ] 有適當的章節分割（5-8 個主要章節）
- [ ] 有實務檢查清單或行動步驟
- [ ] 有延伸閱讀的 Wikilink

#### 代碼和語法
- [ ] Frontmatter 完整（title, description, tags, level, crop, tech 等）
- [ ] 代碼塊正確使用（```）
- [ ] 表格正確格式化
- [ ] Wikilink 正確（[[文章標題]]）

#### 案例和證據
- [ ] 有實際案例或農友故事（去識別化後）
- [ ] 有數據支持（成本分析、收入對比等）
- [ ] 有圖表或視覺化（表格、流程圖）

**品質評分**：_____/10

---

### 第 5 步：決定行動（Action Decision）

根據第 2 步的分類對齊結果，決定：

#### 情況 A：「補充現有文章」（推薦）

```
農友問題 → 已有相關文章
           → 在該文章中添加「農友案例」段落
           → 使用農友的去識別化故事作為例子
```

**行動**：
- [ ] 找出目標文章
- [ ] 在案例/參考段落中補充農友故事
- [ ] 更新文章的最後修改日期
- [ ] 重新生成搜尋索引

**何時適用**：
- 小番茄農友 × 已有「小番茄栽培技術」
- 想直銷的農友 × 已有「直銷客戶開發實務」
- 想建品牌的農友 × 已有「品牌建立與精品化策略」

#### 情況 B：「創建新文章」（謹慎使用）

```
農友問題 → 新作物或新主題
         → 沒有相關文章
         → 符合「新文章創建標準」
         → 創建新文章
```

**創建新文章前必須檢查**：
- [ ] 是否確實是新作物/新主題？
- [ ] 是否已在其他分類下有類似內容？
- [ ] 這篇新文章是否有其他農友也會需要？
- [ ] 內容是否足夠深入（至少 2,000 字以上）？

**新文章標準**：
- 最少 2,000 字
- 包含 3+ 農友案例或實務經驗
- 有完整的檢查清單或實施步驟
- 有 Wikilink 連接到相關文章

#### 情況 C：「暫不處理」（保存待決）

```
農友的問題過於特殊或不符合「教育知識」定位
  → 保存在「待處理清單」
  → 定期檢查是否有更多農友有類似問題
  → 當 3+ 農友有同樣問題時，才考慮創建文章
```

**行動**：
- [ ] 保存農友資料至 `/source/待決/`
- [ ] 記錄問題的通用描述
- [ ] 等待累積足夠案例

---

### 第 6 步：最終發布（Publication）

**發布前最終檢查清單**：

格式和技術
- [ ] 已通過去識別化驗證
- [ ] Frontmatter 完整正確
- [ ] Wikilink 和連結都正確
- [ ] 無拼寫或語法錯誤
- [ ] 測試本地預覽（`npm run dev`）

內容對齊
- [ ] 已確認分類正確
- [ ] 內容與分類定義一致
- [ ] 與其他文章無明顯重複

SEO 和可發現性
- [ ] 標題包含關鍵作物/技術名稱
- [ ] 描述包含農友可能搜尋的詞
- [ ] 添加適當的 tags

**發布步驟**：
1. 提交至 Git：`git add knowledge/[Category]/[Article].md`
2. 提交 Commit：`git commit -m "content: add/update [Article Title]"`
3. 運行同步：`npm run sync`
4. 運行構建：`npm run build`
5. 推送遠端：`git push origin main`

## ⚠️ 常見陷阱與對策

### 陷阱 1：農友故事太獨特，難以去識別化

**症狀**：農友的經歷包含很多個人細節，移除後文章變得無趣

**對策**：
- 保留農業側面的細節，移除個人側面
- 將「Farmer-D從工程師返鄉改行種小黃瓜」改為「設施小黃瓜農場經營者」
- 保留「他如何解決技術問題」，不保留「他怎麼和家人協商」

### 陷阱 2：過度簡化，失去案例的說服力

**症狀**：去識別化後，案例變得太空泛，讀者無法産生共鳴

**對策**：
- 保留具體的數字（成本、收入、產量）
- 保留具體的決策邏輯（為什麼選這個通路）
- 保留失敗和教訓（才有說教意義）

### 陷阱 3：創建太多細微主題的文章

**症狀**：每個農友一篇文章，最後知識庫變得碎片化

**對策**：
- 優先「補充現有文章」而非「創建新文章」
- 設定「3 農友規則」：只有 3+ 農友有同樣問題時，才考慮新文章
- 定期檢查內容重複率（>50% 重複 = 應該合併）

### 陷阱 4：忽視分類對齊，導致文章成孤島

**症狀**：新文章沒有 Wikilink 連到其他文章，讀者難以發現

**對策**：
- 創建文章前，先檢查其他 2-3 篇相關文章
- 在新文章的「延伸閱讀」中加入現有文章
- 同時在現有文章中添加反向 Wikilink
