# 農友資料去識別化政策與檢查清單

**生效日期**：2026-04-20
**最後更新**：2026-04-28
**版本**：1.1（整合驗證檢查清單）
**適用範圍**：所有涉及農友數據的開發、撰寫、提交工作

> 本文件是專案唯一的去識別化單一真相來源。涵蓋政策（什麼能/不能）、分級規則（保留 vs 移除）、與檢查清單（如何驗證）。
> 操作流程請見 [docs/FARMER-DATA-WORKFLOW.md](docs/FARMER-DATA-WORKFLOW.md)。

---

## 0. 核心原則

```
保護個人隱私  ＞  保留學習價值  ＞  最大化內容廣度
```

每篇文章的目標：**讀者能從文章學到技術，但無法推測出具體農民。**

疑問時：**寧刪勿留**。

## 1. 嚴禁事項

### 1.1 提交訊息（commit message）中

- ❌ 農友真實姓名
- ❌ 農場或企業名稱
- ❌ 具體驗證報告檔名（如 `VERIFICATION-REPORT-2.1.md`）
- ❌ 電話、地址、社交媒體帳號

**範例：**
- ❌ `docs: add VERIFICATION-REPORT-2.1 with specific farmer names in case studies`
- ✅ `docs: add internal farmer case study documentation`

### 1.2 版控追蹤的檔案中

- ❌ `source/` 資料夾（含原始農友資料）
- ❌ 含真實姓名的敏感文件
- ❌ 未去識別化的驗證報告
- ❌ `EXECUTION-SCHEDULE` 等列舉真實姓名的工作文件

### 1.3 程式碼註解中

- ❌ 提及農友姓名或敏感細節
- ❌ 硬編碼農友資料

## 2. 三級分級規則

### 🔴 第 1 級：必須移除（RED ZONE）

**絕對不能**出現在發布版本與版控檔案中。

#### 2.1 個人身份資訊
- 全名、可唯一識別的別名
- 配偶、子女、親戚名字
- 可識別個人的照片

#### 2.2 聯絡資訊
- 電話、Line ID、WeChat ID、Email
- 農場地址（門牌號）
- 社群媒體帳號

#### 2.3 精確地理資訊
- 具體里別（如「○○鄉○○村第○鄰」）
- 農地地號、GPS 座標
- 可反推到個人的農場 / 品牌名稱

#### 2.4 識別性的特殊資訊
- 公開獲獎名單上的紀錄（如「2024 百大青農」若可被搜尋）
- 媒體採訪紀錄
- 過於獨特的商業模式 / 家族故事細節
- 特定課程班別編號（如「第○屆○○班」）

#### 替代範例

| 錯誤 | 正確 |
|---|---|
| 「○○○是阿里山山葵農場主人」 | 「某位阿里山山葵農場經營者」 |
| 「我和我老婆、女兒一起經營農場」 | 「家族經營的農場」 |
| 「可聯繫 0912-345-678」 | 「可透過農會聯繫」 |
| 「○○鄉果樹產銷班第 10 班」 | 「嘉義酪梨產銷班」 |
| 「2024 百大青農得主」 | 「青農代表」或「優秀農民」 |
| 「曾是高級廠務工程師，後返鄉務農」 | 「具有工程背景的農場經營者」 |

### 🟡 第 2 級：謹慎處理（YELLOW ZONE）

可保留，但必須加工，確保無法反推到個人。

#### 2.5 財務資料

**原則**：保留比例與趨勢，模糊化絕對數字。

✅ 保留：
- 「年收入約 100-150 萬」（範圍）
- 「毛利率約 40-50%」
- 「成本結構：肥料 30%、人工 40%、其他 30%」（比例）

❌ 不保留：
- 「年收入 1,234,567 元」（精確值）
- 「2023 年賺 150 萬，2024 年 200 萬」（年度精確對應）

#### 2.6 規模資料

**原則**：可保留，但四捨五入到合理範圍。

✅ 保留：「0.35 公頃」「年產 75 噸」
❌ 過度精確：「0.3467 公頃」→ 改「約 0.35 公頃」；「3,847 株」→ 改「約 3,800-4,000 株」

#### 2.7 特殊技術或設備

**原則**：可描述技術，不提品牌或供應商（除非為公開商用通用產品）。

✅ 保留：「導入智慧灌溉系統」「使用養液栽培」
❌ 修改：「採用 Autopot 系統」→「採用進口自動潮汐灌溉系統」

#### 2.8 家族資訊

**原則**：可提家族經營，不涉個人生活故事。

✅ 保留：「家族農場，三代接力」「家族成員各司其職」
❌ 不保留：「兒子是台大農業系畢業」「父親因過勞健康惡化」

### 🟢 第 3 級：可完全保留（GREEN ZONE）

#### 2.9 農業技術
品種名、栽培方式、管理技巧、病蟲害診斷。

#### 2.10 市場與銷售
通路類型、客戶類型、定價策略、市場趨勢。

#### 2.11 經營管理
組織結構、管理流程、決策邏輯、風險管理。

#### 2.12 數據與統計
市場行情、成本結構（百分比）、回報率、時間線（年數而非具體年份）。

## 3. 反向工程測試

每篇文章發布前，閉上眼睛，假裝是網路上的陌生人讀完文章，問自己：

| 問題 | 通過標準 |
|---|---|
| 我能從這篇文章推測出「具體農民」嗎？ | ✅ 不能 |
| 農民親戚 / 鄰居能認出他嗎？ | ✅ 不能（除非親身經歷者） |
| 文章是否有「2-3 農友」的集合特徵？ | ✅ 是 |

**任何一題不通過 → 進一步模糊化或刪除**。

特別檢查能否透過以下方式反推：
1. 對比公開的百大青農名單
2. 搜尋新聞報導中的農民
3. 用「嘉義 + 作物 + 收入區間」搜尋
4. 認出獨特的故事細節（如「工程師返鄉」）
5. 認出特殊商標 / 品牌

## 4. 提交前檢查清單

### 4.1 直接識別符（檢查清單 A）

- [ ] 文中無農友真實姓名（搜尋 "姓名"、農友名）
- [ ] 通用代稱為「某農友」「農友 A / B / C」或「某設施農民」
- [ ] 無完整郵編、村里號（搜尋 "里"、"村"、"號"、"路"、"鄰"）
- [ ] 地理資訊最多到「鄉/鎮」級
- [ ] 無電話、Email（搜尋 `@`、"電話"、"手機"）
- [ ] 無社群帳號（搜尋 "Line"、"Facebook"、"IG"）

### 4.2 間接識別符（檢查清單 B）

- [ ] 無家族成員名字（搜尋 "太太"、"老婆"、"先生"、"媽媽"、"父"、"子"）
- [ ] 獨特工作背景已模糊化（搜尋 "原本"、"曾經"、"之前"、"背景"）
- [ ] 特殊設備品牌已替換為通用描述
- [ ] 無特定課程 / 班別編號（搜尋 "第"、"屆"）

### 4.3 反向工程測試（檢查清單 C）

- [ ] 能否推測具體農民？— 否
- [ ] 親友能否認出？— 低風險
- [ ] 文中混合 ≥ 2 位農友的集合特徵 — 是

### 4.4 提交前 git 自查

```bash
# 查看即將提交的內容
git diff --cached

# 搜尋潛在敏感詞
git diff --cached | grep -iE "姓名|電話|地址|source/"

# 確認 source/ 沒被加入
git status | grep -i source/
```

只有以上全部通過，才可 `git commit`。

## 5. `.gitignore` 必含項

以下規則已在 [.gitignore](.gitignore)（標示為「FARMER DATA PROTECTION」段）：

```gitignore

# 敏感驗證與追蹤文件
docs/VERIFICATION-REPORT-*.md

# 私密設定與憑證
.env.local
.credentials/

# 備份檔案
*.json.bak
*.md.bak
```

**不要刪除或修改這段規則**。

## 7. 違反規範後的處理

如不慎提交了敏感資料：

1. **立即停止** — 不要再 push 或繼續提交
2. **歷史清理** — 用 `git-filter-repo` 從歷史移除
3. **強制推送** — 覆寫 GitHub 上的歷史
4. **通知相關人員** — 確保敏感資料已被清理
5. **更新本政策** — 防止未來重複

## 8. 例外情況：何時可保留農民姓名

**僅在以下情況**才可保留農民真實姓名：

1. ✅ **農民書面同意** — 主動要求署名
2. ✅ **已公開身份** — 農民的農場品牌已在新聞、百大青農等公開渠道發表
3. ✅ **學術研究** — 已簽署知情同意書（Informed Consent）

即使滿足以上，仍應：

- 取得明確書面同意
- 在文章開頭註明「本文經農民授權發表」
- 保留撤銷權（若農民後來要求移除，必須照辦）

## 9. 自動化工具

| 工具 | 路徑 | 用途 |
|---|---|---|
| pre-commit hook | [.husky/pre-commit](.husky/pre-commit) | 提交前去識別化檢查 + frontmatter + wikilink 驗證 |
| commit-msg hook | [.husky/commit-msg](.husky/commit-msg) | 提交訊息雜湊式去識別化檢查 |
| 批量替換（bash） | [scripts/tools/deidentify-source.sh](scripts/tools/deidentify-source.sh) | 處理 `source/` 中的已知敏感詞 |
| 批量替換（Python） | [scripts/tools/deidentify-source.py](scripts/tools/deidentify-source.py) | 雜湊比對 + 進階替換 |
| 加新敏感詞 | [scripts/tools/add-sensitive-name.sh](scripts/tools/add-sensitive-name.sh) | 動態擴充黑名單 |

> ⚠️ hook 必須有執行權限才會生效：
> ```bash
> chmod +x .husky/pre-commit .husky/commit-msg
> ls -l .husky/pre-commit .husky/commit-msg   # 應顯示 x 權限
> ```

## 10. 不確定時的決策樹

```
遇到一句話，不確定是否合規？

問 1：這句話是否包含「可直接識別的資訊」？
  → 是：刪除或改為通用描述
  → 否：問 2

問 2：這句話結合其他資訊，是否「幫助認出農友」？
  → 是：刪除或進一步模糊化
  → 否：問 3

問 3：讀者是否必須知道這句話才能學到技術？
  → 是：保留，但嘗試改為通用描述
  → 否：刪除

最終原則：
  有疑問 → 刪除或模糊化
  沒疑問 → 保留
```
